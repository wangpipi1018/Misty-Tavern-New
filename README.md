🍻 迷霧酒館 (Misty Tavern) 開發者手冊本專案是一個基於 Web 的高複雜度骰子策略對戰遊戲，支援 單機 PvE 與 Firebase 聯網 PvP。採用「數據、邏輯、視圖」分離的模組化架構，以維護具備大量被動連鎖（Chain Effects）的對戰系統。🏗️ 檔案架構與核心職責1. index.html (主入口與異步監聽層)職責：DOM 結構、Firebase 監聽（listenToBattleActions）、聯網指令中轉站（BattleCommander）。關鍵機制：所有聯網動作透過 lastAction 進行廣播，由 BattleCommander.dispatch 執行。2. game-state.js (數據層)核心變數：window.gs (Global State)。職責：儲存戰鬥瞬間的真理數值（血量、能量、翻面次數、配件帳本）。初始化：透過 createBattleState 建立，包含聯網所需的 currentTurnPlayer 標記。3. battle-fx.js (原子邏輯層)核心物件：window.FX。職責：定義所有「會改變數據」的原子動作（damage, heal, stealPts）。同步規範：此層負責觸發被動事件廣播（如 steal, revive），確保聯網兩端數值對齊。4. accessories-config.js (配件配置層)核心變數：window.ACCESSORIES。職責：定義配件的觸發時機（on 事件）與效果。規範：禁止直接修改 gs，必須呼叫 FX 提供的函式以維持同步路徑。5. ui-render.js & ui-interaction.js (視圖與互動層)職責：處理拖曳骰子的物理反饋與戰鬥畫面的視覺刷新（renderBattle）。🛡️ 核心同步機制：權威事件流 (New)為了應對網路延遲與被動連鎖，本專案捨棄了簡單的數據覆寫，改用 「受害者權威廣播」 模式：傷害攔截：damage 函式不分對象執行，但只有「受害者端」有權發起 revive (不死鳥) 或 steal (反甲) 廣播。勝利攔截：在 checkWin 中加入「不死鳥檢查」，若對手血量歸零但具備重生資格，會暫緩宣告勝利，直到重生封包到達。能量守恆：透過 steal 事件同步點數增量，而非同步絕對值，避免覆蓋掉玩家本地正在進行的操作。🛠️ 函式索引表 (功能分類)⚔️ 戰鬥核心邏輯 (Battle Core)函式名稱職責說明檔案位置doRoll執行擲骰，處理 bonusDice 與 nextDie
